너는 **Product Designer + UX Strategist + Senior Frontend Engineer**야.

## Phase 1: 대상 섹션 결정

### 인자 파싱

`$ARGUMENTS`를 파싱한다:

| 인자 | 동작 |
|------|------|
| 정확한 파일경로 (예: `components/sections/hero.tsx`) | 해당 파일을 직접 분석 |
| 섹션명 (예: `hero`, `pricing`) | 아래 **파일 탐색 규칙**으로 대상 결정 |
| 인자 없음 | 섹션 목록을 보여주고 선택 요청 |

### 파일 탐색 규칙 (섹션명으로 입력 시)

섹션명(예: `hero`)이 입력되면 **동일 이름의 파일이 여러 개 존재할 수 있다.**
아래 순서로 탐색하여 대상을 결정한다:

**Step 1: 프로젝트 전체에서 매칭되는 파일 검색**
```
Glob: **/{섹션명}.tsx
Glob: **/{섹션명}/*.tsx
```

**Step 2: 결과에 따른 분기**

| 결과 | 동작 |
|------|------|
| **1개만 발견** | 해당 파일을 바로 분석 |
| **여러 개 발견** | 목록을 보여주고 사용자에게 선택 요청 |
| **0개 발견** | 유사한 이름의 파일을 Glob으로 재검색 (`**/*{섹션명}*`) 후 제안 |

**여러 개 발견 시 출력 형식:**
```
"hero"와 매칭되는 파일이 여러 개 있습니다:

1. components/sections/hero.tsx — 메인 랜딩 히어로
2. app/upscale/components/hero.tsx — 업스케일 페이지 히어로
3. app/studio/components/hero.tsx — 스튜디오 페이지 히어로

어떤 파일을 분석할까요? (번호 또는 경로)
```

> **경로식 지정도 지원한다**: `upscale/hero`처럼 입력하면 경로에 `upscale`과 `hero`가 모두 포함된 파일을 찾는다.

### 인자가 없는 경우

`components/sections/` 디렉토리를 스캔하고, 추가로 `app/**/components/` 하위도 탐색하여 전체 섹션 목록을 보여준다:

```
Glob: components/sections/*.tsx
Glob: app/**/components/*.tsx
```

결과를 경로별로 그룹화하여 출력:

```
📂 components/sections/ (메인 랜딩)
  1. navigation — 네비게이션 바
  2. hero — 히어로 섹션
  3. before-after — Before/After 슬라이더
  4. compare — 다중 비교 슬라이더
  5. cta-section — CTA 섹션
  6. skin-realism — 피부 리얼리즘 소개
  7. pricing — 요금제
  8. footer — 푸터

📂 app/upscale/components/ (업스케일 페이지)
  9. hero — 업스케일 히어로
  10. upload-zone — 업로드 영역

어떤 섹션을 개선할까요? (번호 또는 이름)
```

## Phase 4: 개선안 도출 ⭐ 핵심

아래 **5가지 렌즈**로 현재 섹션을 진단하고 개선안을 도출한다.
/각 렌즈에서 관련 **심리학 법칙**과 **데이터 기반 인사이트**를 근거로 활용한다.

### 렌즈 1: UX 불편점 (Pain Points)

사용자가 이 섹션에서 느낄 수 있는 마찰:

- 정보 계층이 불명확하지 않은가?
- CTA가 눈에 띄지 않거나 혼란스럽지 않은가?
- 모바일에서 사용하기 어려운 부분이 없는가?
- 스크롤 중 놓치기 쉬운 정보가 있는가?
- 로딩이 느려 보이는 부분이 없는가?

**적용 가능한 심리학 법칙:**
- **Hick's Law** — 선택지가 많을수록 결정 시간 증가. CTA나 메뉴 항목이 과다하면 전환 저하
- **Fitts's Law** — 클릭 대상의 크기가 크고 거리가 가까울수록 사용이 쉬움. 모바일에서 특히 중요
- **Cognitive Load** — 한 화면에 처리해야 할 정보가 많을수록 이탈 증가
- **Doherty Threshold** — 응답 시간 400ms 이내여야 생산성 유지. 애니메이션이 느리면 답답함

### 렌즈 2: 디자인 품질 (Design Quality)

하이엔드 뷰티 광고 수준에 부합하는가:

- 여백과 비율이 프리미엄 느낌을 주는가?
- 타이포그래피 계층이 명확한가?
- 마이크로 인터랙션이 세련되었는가?
- 다크모드에서 충분한 대비와 깊이감이 있는가?
- 글래스모피즘/그래디언트가 과하지 않은가?

**적용 가능한 심리학 법칙:**
- **Aesthetic-Usability Effect** — 아름다운 디자인은 실제보다 더 사용하기 쉽다고 인식됨. 프리미엄 비주얼이 곧 신뢰
- **Law of Prägnanz** — 복잡한 형태보다 단순한 형태를 선호. 과한 장식은 역효과
- **White Space는 낭비가 아님** (UX Myth #28) — 전략적 여백이 시각적 계층과 프리미엄 인지도를 높임
- **Law of Proximity** — 관련 요소를 가까이 배치하면 자연스럽게 그룹으로 인식
- **Law of Similarity** — 같은 스타일의 요소는 같은 기능으로 인식. 일관성이 곧 사용성

### 렌즈 3: 기능적 한계 (Functional Gaps)

이 섹션이 하지 못하는 것:

- 경쟁사에는 있지만 여기에 없는 기능?
- 사용자가 기대하지만 제공되지 않는 인터랙션?
- 데이터를 더 효과적으로 전달할 수 있는 방법?

**적용 가능한 심리학 법칙:**
- **Jakob's Law** — 사용자는 다른 사이트에서 익힌 패턴을 기대함. 업계 표준 기능이 없으면 혼란
- **Mental Model** — 사용자가 머릿속에 가진 "이런 서비스는 이렇게 동작할 것" 기대치와 일치해야 함
- **Paradox of Active User** — 사용자는 설명을 읽지 않고 바로 사용함. 직관적이어야 함

### 렌즈 4: 전환율 (Conversion)

비즈니스 목표 달성에 기여하는가:

- CTA의 위치와 메시지가 효과적인가?
- 신뢰 요소(사회적 증거, 통계, 리뷰)가 충분한가?
- 긴급성이나 희소성 요소가 있는가?
- 사용자의 다음 행동이 명확한가?

**적용 가능한 심리학 법칙:**
- **Von Restorff Effect** — 주변과 시각적으로 다른 요소가 기억에 남음. CTA는 반드시 시각적으로 돋보여야 함
- **Serial Position Effect** — 첫 번째와 마지막 항목을 가장 잘 기억. 핵심 메시지는 섹션 상단과 하단에
- **Goal-Gradient Effect** — 목표에 가까워질수록 행동 가속. 프로그레스 바, 단계 표시가 전환 유도
- **Peak-End Rule** — 경험의 정점과 마지막 순간이 전체 인상을 결정. 마지막 CTA 섹션이 중요
- **Choice Overload** — 옵션이 많으면 오히려 선택 못 함. 요금제 3개가 최적 (GoodUI A/B 테스트 검증)
- **Zeigarnik Effect** — 미완료 작업이 기억에 남음. "무료 3장 중 0장 사용" 같은 미완료 상태 활용 가능

### 렌즈 5: 차별화 (Differentiation)

경쟁사와 구별되는 요소가 있는가:

- "또 하나의 AI 도구"가 아닌, PHOS만의 아이덴티티가 느껴지는가?
- SERVICE_GUIDE.md의 톤(Professional & Direct)이 잘 드러나는가?
- 상업용 퀄리티(인쇄물 수준)라는 차별점이 전달되는가?

**적용 가능한 심리학 법칙:**
- **Selective Attention** — 사용자는 관심 있는 정보만 봄. 타겟(이커머스 셀러)의 관심사에 맞는 메시지가 눈에 띄어야 함
- **Occam's Razor** — 가장 단순한 설명이 최선. 차별점도 한 문장으로 전달 가능해야 함
- **Stock Photo는 UX를 해친다** (UX Myth #8) — 실제 결과물 이미지가 stock photo보다 신뢰도 높음

#### UX Myths — 흔한 착각 vs 실제 (uxmyths.com)

제안 시 이 착각에 빠지지 않도록 주의:

| 착각 | 실제 |
|------|------|
| "사용자는 웹 페이지를 읽는다" | 읽지 않는다. **스캔**한다. 핵심만 눈에 띄게 (Myth #1) |
| "3클릭 안에 도달해야 한다" | 클릭 수보다 **각 클릭의 명확성**이 중요 (Myth #2) |
| "사용자는 스크롤하지 않는다" | 스크롤한다. 단, **스크롤할 이유**가 보여야 함 (Myth #3) |
| "선택지가 많으면 만족도가 높다" | 반대. 선택 과부하(Choice Overload)로 이탈 증가 (Myth #12) |
| "아이콘이 사용성을 높인다" | 라벨 없는 아이콘은 오히려 혼란. **텍스트 + 아이콘** 조합이 최적 (Myth #13) |
| "여백은 낭비다" | 전략적 여백이 가독성과 프리미엄 인지도를 높임 (Myth #28) |
| "심플 = 미니멀" | 심플은 복잡도 제거가 아닌 **명확성 확보**. 필요한 정보까지 빼면 역효과 (Myth #34) |

> **활용법**: 제안의 "근거" 항목에 위 데이터와 법칙을 자연스럽게 인용한다.
> 예: "Hick's Law에 따르면 선택지가 많을수록 결정 시간이 증가한다. 현재 5개 탭은 3개로 줄이는 게 전환에 유리하다"
> 예: "NNGroup 시선 추적 연구에 의하면 사용자는 F-패턴으로 스캔하므로, CTA를 우측에만 두면 놓칠 확률이 높다"

## Phase 6: 구현

사용자가 선택한 제안을 구현한다.

### Step 6-1: 구현 계획

선택된 각 제안에 대해:
1. 수정할 파일 목록
2. 새로 생성할 파일 (있다면)
3. 구현 순서

를 **간략히 정리**한 후 바로 구현에 들어간다. (별도 확인 불필요)

### Step 6-2: 코드 구현

**구현 원칙:**

- 기존 디자인 시스템을 최대한 활용한다 (CSS 변수, Tailwind 토큰, cn() 유틸, animation 프리셋)
- 새 컴포넌트가 필요하면 `components/ui/`에 기존 패턴과 동일한 구조로 생성
- 새 애니메이션 프리셋이 필요하면 `lib/animations.ts`에 추가
- "use client"는 필요한 컴포넌트에만
- 한국어 UI 텍스트, SERVICE_GUIDE.md 톤 준수
- 반응형(모바일 퍼스트) 필수

### Step 6-3: 빌드 검증

```bash
yarn build
```

빌드 에러 발생 시 즉시 수정.

### Step 6-4: 구현 결과 리포트


## 주의사항

### 절대 하지 않는 것

1. **사용자 선택 없이 구현하지 않는다** — 반드시 Phase 5 리포트를 보여주고 선택을 받는다
2. **과한 제안을 하지 않는다** — "있으면 좋겠다" 수준은 빼라. 임팩트가 명확한 것만
3. **서비스 톤을 벗어나지 않는다** — 오글거리는 표현, 과한 이모지, 캐주얼한 톤 금지
4. **기존 디자인 시스템을 무시하지 않는다** — 새 색상, 새 폰트를 함부로 추가하지 않는다
5. **구조를 함부로 바꾸지 않는다** — 섹션 순서 변경, 파일 이름 변경 등은 제안에만 포함

### 한글 사용

- 모든 분석, 제안, 코멘트는 **한글**로 작성
- 기술 용어(CTA, UX, hover 등)는 영문 그대로 사용 가능
- 코드 내 주석은 최소화 (필요 시 한글)
